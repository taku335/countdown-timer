# 設計仕様（概要）

## 1. 目的
- タイマーアプリを提供し、静的サイトを用意する。
- GitHub Pages 上で公開する前提を維持し、ビルド成果物のみを配信対象とする。

## 2. 対象ユーザー
- 会議等の場で時間計測をする社内メンバー
- UI/UX の確認を行うステークホルダー

## 3. スコープ
### 含む
- React + Vite による UI 構築
- 時刻通知ビューと入力フォーム
- ハッシュルーティングによる画面遷移
- ビルド工程による成果物の生成と配信

### 含まない
- 実検索処理（外部データ取得や永続化）
- サーバーサイド処理、外部 API 連携
- 認証やユーザー管理

## 4. 画面構成
### 4.1 ルーティング
- HashRouter による3画面構成。
  - 会議室検索フォーム（`#/`）
  - 研修用時刻通知ページ（`#/time`）
  - 404（その他のパス）

### 4.2 画面: 研修用時刻通知
- 構成: 表示エリア（現在時刻・終了時刻・残り時間）、入力フォーム
- 入力項目
  - タイトル、終了時刻、終了時の配色
  - クイック設定（10 / 15 / 30 / 60 分）
- 操作
  - 残り時間を1秒ごとに更新

## 5. 画面仕様（UI/UX）
- 読みやすい和文サンセリフ系のフォント。
- 背景とカードに階層を作り、入力に集中しやすい視認性を確保。
- 1000px前後の中央レイアウトを基本とし、レスポンシブに1カラムへ切り替える。
- デモであることを補足テキストで明示する。

## 6. 機能仕様
- 入力はブラウザ標準のフォーム要素を使用。
- 時刻通知はクライアント側の時計のみで表示。
- 画面遷移はハッシュルーティングで完結。

## 7. 技術構成
- **Build Mode** を採用する。
- フレームワーク: React + Vite。
- ディレクトリ構成
  - `src/`: ソース
  - `dist/`: ビルド成果物
- 画像などのアセットは必要最小限に留める。
- 外部 API 連携や外部 SDK の読み込みは行わない。

## 8. デプロイ/パス設計
- GitHub Pages では `dist/` を公開対象とする。
- `.github/workflows/deploy-pages.yml` の `Upload artifact` は `dist` を指定する。
- ルーティングはハッシュ形式のため、追加の `404.html` フォールバックは不要。
- アセット参照およびリンクは相対パスを基本とする。

## 9. 制約・注意事項
- サーバーサイド処理は不可。
- 機密情報・APIキー等は追加しない。
- 公開物は `dist/` のみを対象にする。

## 10. 今後の拡張方針（検討レベル）
- 画面追加はルーター定義で管理し、必要に応じて静的データ表示を追加する。
- 検索結果表示を拡張する場合も、クライアント内の静的データで完結させる。
